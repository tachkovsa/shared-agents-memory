# ArtelMemory — Brand & Landing Direction for Claude Design

## 1. Product Context

**Product name:** ArtelMemory  
**Domain:** artelmemory  
**Current repository rename target:** from `shared-agents-memory` to `artelmemory` or `artelmemory-core`.

ArtelMemory is a shared memory layer for AI agents. It is similar in spirit to products like `claude-mem`, but with a stronger focus on self-hosted deployments, local infrastructure, CIS markets, and teams that want to keep their data under control.

The product gives AI agents a persistent, shared, queryable context across tools, sessions, and workflows.

Primary users:
- developers using Claude Code, Codex, OpenCode, Cursor-like tools, and internal AI agents;
- small engineering teams using several AI agents in parallel;
- companies in CIS markets that care about data residency, local hosting, and local/regional LLM providers;
- organizations that want AI agents but cannot freely send project context to external cloud providers.

Core idea:

> One shared memory for your AI agents.

Russian-facing landing message:

> Единая память для ваших ИИ-агентов.

English equivalent:

> Unified memory for your AI agents.

The brand should feel like reliable infrastructure, not a trendy AI toy.

---

## 2. Brand Positioning

ArtelMemory should be positioned as:

> A self-hosted memory infrastructure layer that lets AI agents remember, collaborate, and reuse project knowledge safely inside the user's own environment.

Key associations:
- shared memory;
- connected agents;
- modular infrastructure;
- local deployment;
- team knowledge;
- trust and data control;
- open-source / developer-first;
- CIS-friendly, but not provincial or overly ethnic.

The word **Artel** refers to collective work, cooperation, and a group of skilled people working together. This should influence the brand subtly: the system should feel collaborative, practical, and grounded.

Avoid making the visual identity too folkloric. The brand can borrow warmth from Central Asia / CIS through color and texture, but it should stay modern, technical, and globally understandable.

## 3. Visual Direction

The preferred visual direction is based on **Concept 7** from the latest exploration:

**Concept:** Modular stack + connected agents.

The logo symbol should communicate:
- memory as a layered stack;
- several AI agents connected to the same memory layer;
- semantic retrieval / vector memory;
- infrastructure that can be deployed locally;
- a clean, mature technical product.

The icon can be built from:
- 2–3 stacked rounded diamond/square layers;
- 2–3 connector lines ending in circular nodes;
- one warm accent element inside or near the stack to represent the shared core/context;
- simplified geometry that works in small sizes.

Avoid:
- overly complex graph networks;
- generic AI sparkles as the main identity;
- too many colors inside the logo;
- mascot-like or playful shapes;
- blockchain-looking logos;
- excessive gradients.

The logo should look good in:
- full horizontal lockup;
- compact header logo;
- GitHub avatar;
- app tile / favicon;
- monochrome version;
- dark background version.

## 4. Logo / Wordmark Recommendations

Explore these wordmark placements:

### Option A — Single word

`ArtelMemory`

Use when the product should feel like a single technical product name. This is compact and good for GitHub, CLI, package names, docs, and developer tools.

Recommended treatment:
- `Artel` in dark graphite;
- `Memory` in muted teal;
- no space;
- clean and modern.

### Option B — Two-word brand

`Artel Memory`

Use when we want stronger readability and a more mature brand feel. This is likely better for landing pages, enterprise decks, and sales materials.

Recommended treatment:
- visible space between `Artel` and `Memory`;
- both words on one line in the main header;
- `Memory` can be teal, while `Artel` stays graphite.

### Option C — Icon as connector between words

`Artel [icon] Memory`

This is more expressive and brandable. The icon becomes the shared memory layer between the two semantic blocks.

Use carefully. It can be strong for hero sections and brand boards, but may be too wide for navigation headers.

### Preferred practical system

Use both:
- **ArtelMemory** as the product/dev name;
- **Artel Memory** as the visual wordmark on marketing pages if it improves readability.

This lets the product remain technically consistent while allowing more elegant branding.

## 5. Color Palette

The previous bright blue SaaS direction felt too generic/global. The new direction should feel warmer, more grounded, and more suitable for CIS / Central Asia while still being technical and enterprise-ready.

### Primary Palette

| Role | Name | HEX | Usage |
|---|---|---:|---|
| Primary dark | Graphite | `#1C2430` | Main text, dark hero, logo base, footer |
| Brand accent | Teal | `#2B7D7A` | Logo accent, primary CTA, highlights, active states |
| Warm accent | Terracotta | `#C96A4A` | Nodes, memory core, badges, small highlights |
| Background | Warm Cream | `#F3EEE4` | Page background, brand boards, soft sections |
| Secondary neutral | Stone Grey | `#D6D2C4` | Logo layer, borders, muted surfaces |
| Text muted | Warm Slate | `#6B7280` | Secondary text, captions |
| Optional utility blue | Functional Blue | `#2A6CF0` | Only for links/focus states if needed, not a main brand color |

### Color Principles

- Teal should be the main brand accent, not bright blue.
- Terracotta should be used sparingly as a warm “shared context” marker.
- Graphite gives enterprise trust and infrastructure depth.
- Warm Cream prevents the product from looking like a generic cold SaaS tool.
- Blue can exist as a UI utility color, but it should not dominate the brand.

### Recommended Logo Colors

- Stack bottom layer: Graphite `#1C2430`
- Stack middle layer: Stone Grey `#D6D2C4`
- Stack top layer: Teal `#2B7D7A`
- Connector lines/nodes: Graphite + Teal
- Shared core / one connector node: Terracotta `#C96A4A`

## 6. Typography

The product needs typography that supports both English and Russian well.

### Recommended Pairing

**Headings / Logo / Hero:** `Sora SemiBold` or `Manrope Bold`  
**Body / UI / Documentation:** `Inter Regular / Medium`

### Why

- `Sora` feels technical, distinctive, and modern.
- `Manrope` has excellent Cyrillic support and feels friendly but still professional.
- `Inter` is reliable for UI, dashboards, forms, docs, and dense technical layouts.

### Preferred system

Use:
- **Manrope** for headings if the landing is primarily Russian/CIS-focused.
- **Inter** for body and UI.

Alternative:
- **Sora** for logo/hero only, **Inter** for all other UI.

Avoid:
- overly geometric fonts with weak Cyrillic;
- serif fonts;
- decorative fonts;
- fonts that make the product look like a crypto/Web3 brand.

## 7. Landing Page Direction

The landing page should be in Russian at first, because Russian is the most practical shared language for the initial CIS audience.

The landing should not look like a generic US/EU AI SaaS page. It should communicate:
- local deployment;
- data control;
- engineering reliability;
- compatibility with local/regional models;
- open-source/self-hosted nature;
- shared memory for AI agents.

### Hero Section

Main headline:

> Единая память для ваших ИИ-агентов

Alternative:

> Память для ваших AI-агентов — в вашем контуре

Subheadline:

> ArtelMemory сохраняет общий контекст проектов, решений и задач между Claude Code, Codex, OpenCode, внутренними copilot-сервисами и другими AI-агентами. Разворачивайте в своей инфраструктуре, подключайте локальные embedding-модели и контролируйте, какие данные куда передаются.

Primary CTA:

> Начать развёртывание

Secondary CTA:

> Посмотреть архитектуру

Header navigation:
- Возможности
- Архитектура
- Развёртывание
- Интеграции
- Документация
- GitHub

Hero visual:
- dark graphite hero card;
- central modular memory stack icon;
- connected agent nodes around it;
- small labels: Claude Code, Codex, OpenCode, Telegram Bot, Internal Copilot;
- deployment badges: Self-hosted, Qdrant, SQLite, Local embeddings, YandexGPT, GigaChat.

### Key Landing Blocks

1. **Problem**
   - Agents forget context between sessions.
   - Knowledge is scattered across GitHub, docs, issues, chats, and local files.
   - Parallel agents duplicate work.
   - Companies cannot freely send context to external providers.

2. **Solution**
   - ArtelMemory becomes the shared memory layer between AI agents.
   - Agents read and write structured memory.
   - Context is retrieved semantically through Qdrant.
   - Embeddings can be generated locally.

3. **Architecture**

   Suggested diagram:

   `AI agents → ArtelMemory API → Memory pipeline → SQLite metadata + Qdrant vectors + local embeddings → relevant context back to agents`

4. **Local-first Deployment**
   - self-hosted;
   - single Docker Compose setup;
   - SQLite for simple product metadata;
   - Qdrant for semantic memory;
   - optional local embedding service;
   - optional future PostgreSQL profile for enterprise.

5. **CIS / Data Residency**
   - deploy in Russia, Kyrgyzstan, Kazakhstan, Uzbekistan, Tajikistan, or on-prem;
   - configure local-only mode;
   - support YandexGPT, GigaChat, local models, OpenRouter, Ollama/vLLM depending on customer policy;
   - avoid promising full legal compliance; say “deployment profiles help align architecture with local data requirements”.

6. **For Whom**
   - individual AI-native developers;
   - engineering teams;
   - internal platform teams;
   - companies building internal AI copilots;
   - CIS organizations with data control requirements.

## 8. UI Style

### General Mood

- reliable;
- warm;
- engineering-first;
- calm;
- open-source friendly;
- not playful;

### Layout

- teal background `#2B7D7A`;

- subtle hover darkening.

Secondary button:
- transparent or warm cream surface;
- graphite text;
- soft border.

### Icons

Use simple line icons with rounded ends:
- shield = control/security;
- stack = memory layer;
- connected nodes = shared agent context;
- server = local hosting;
- plug/API = integrations;
- graph/search = semantic retrieval.

## 9. Messaging Principles

Use simple, direct Russian copy. Avoid over-marketing and AI buzzword overload.

Good phrases:
- Единая память для ваших ИИ-агентов
- Контекст, который не теряется между сессиями
- Данные остаются в вашем контуре
- Локальные embeddings без передачи документов наружу
- Qdrant для семантического поиска
- SQLite для лёгкого self-hosted запуска
- Подключайте Claude Code, Codex, OpenCode и внутренних агентов
- От одного разработчика до команды и enterprise-контура

Avoid:
- “революционный AI”;
- “магическая память”;
- “самый умный агент”;
- “полное соответствие законам всех стран”;
- “данные никогда никуда не передаются” unless this is strictly true for the selected deployment profile.

Use careful wording:

> ArtelMemory can be deployed in a local-only mode where documents and embeddings remain inside your infrastructure.

Instead of:

> ArtelMemory guarantees legal compliance and complete data protection.

## 10. Technical Truths to Reflect in Design

Current architecture assumptions:

- SQLite is used for product metadata: users, projects, agents, settings, permissions, audit, memory metadata.
- Qdrant is used for vectors, semantic search, retrieval, payload filtering, and memory collections.
- Local embeddings are strongly recommended for self-hosted / regulated deployments.
- PostgreSQL can be presented as a future or enterprise profile, not the default.
- The product should support local and regional LLM providers depending on deployment policy.

Recommended technical positioning:

> SQLite for lightweight self-hosted metadata. Qdrant for semantic memory. Local embeddings for safe indexing. Your infrastructure remains the source of truth.
